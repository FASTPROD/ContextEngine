### D. `docs/DEPLOY_ARCHITECTURE_UPGRADE.md` — design doc (commit `bc9ac5a`)

User's 3rd question deserved a real architectural answer. The doc compares 5 patterns:
- **Status quo** — hardened rsync (where we are; B+ safety)
- **Option A — atomic symlink swap** — rsync into NEW timestamped dir, then `ln -sfn`. Sub-second rollback, zero live-dir race. Recommended for compR.fr. ~50 min total to ship.
- **Option B — blue/green pm2 + nginx upstream** — zero-downtime deploys, crash-safe rollback. Recommended for api.compr.ch. ~75 min total. **But cross-app blast radius on konive-ovh needs multi-agent diagnostic FIRST.**
- **Option C — staging subdomain** — low-effort safety multiplier, stacks with A or B
- **Option D — pull-based deploy** — eliminates SSH credentials on dev machine, useful for multi-dev teams
- **Option E — full CI/CD** — heaviest, production-grade

User picked: **A first, then B (with multi-agent diagnostic before touching konive-ovh)**.

### E. Option A — compR.fr atomic symlink (SHIPPED, compR.fr commit `727c98a`)

Two new scripts in the `compR.fr` repo (sibling to the existing `deploy.sh` v1):

- **`setup-symlink-deploy.sh`** — ONE-TIME server-side conversion. Converts `/var/www/comprfr` from a regular directory into a symlink → first versioned dir (`/var/www/comprfr-<ts>/`) via hardlink-copy (`cp -al`, nearly free disk). Atomic swap. Auto-rollback if nginx no longer serves 200 post-swap. LOCK `[COMPR-FR-SYMLINK-SETUP]` forbids re-running.
- **`deploy-v2.sh`** — replaces `deploy.sh` v1 for all future deploys. REFUSES to run unless `/var/www/comprfr` is already a symlink (setup must run first). Creates a NEW versioned dir from current target via `cp -al`, rsyncs whitelisted files into the NEW dir (NOT touching the live symlink target), smoke-tests the new dir's `index.html` for `HEALTH_MARKER` BEFORE flipping, then atomic `ln -sfn` flip, post-flip live verify with auto-rollback. `--rollback` interactively picks any prior versioned dir; `--prune` drops dirs older than `KEEP_LAST_N=5`. LOCK `[COMPR-FR-ATOMIC-SYMLINK]`.

**Why safer than deploy.sh v1**: v1 rsyncs into the live serving directory — mid-rsync visitors can see half-old/half-new state. v2 rsyncs into a NEW dir nginx isn't yet serving; the only customer-visible moment is the filesystem-atomic `ln -sfn`. Rollback in v2 = one `ln -sfn` back to previous timestamped dir (sub-second, no rsync needed). v1 + v2 can coexist — v1 stays as fallback.

**Activation:** user runs `./setup-symlink-deploy.sh --really-setup` once (carry-forward to Sprint 15+), then uses `./deploy-v2.sh` for all subsequent deploys.

### F. Option B — multi-agent diagnostic launched (workflow `wdcraou93`, in flight at session-doc commit)

Per user's directive: "be careful this server has many more apps! should you run multi-agent root-cause first?" Launched workflow following `docs/skills/CLAUDE_MULTI_AGENT_PROMPT.md`:

- **Agent A** (Plan subagent, code-audit) — reads `server/ecosystem.config.cjs`, `server/deploy.sh`, `server/src/server.ts`, `server/package.json`, identifies file:line hotspots constraining or enabling the blue/green change, ranks cross-app risk LOW/MED/HIGH, single-sentence verdict naming ONE change with best impact/risk ratio.
- **Agent B** (Plan subagent, adversarial) — independent read of the same source. Deliberately hunts contradicting evidence: pm2 reload semantics (does it restart ALL ecosystem processes, not just opscontext?), nginx shared-upstream blocks (would a syntax error take down sibling apps via `nginx -s reload`?), port collision (are 3000+3001 actually free?), server.ts port-binding semantics (two instances colliding on same default?), shared resources (DB / SSL cert paths), botched-flip blast radius.
- **Foreground (probe-list builder)** — instead of Playwright (no UI), builds a checklist of READ-ONLY ssh commands the user runs tomorrow to gather the runtime facts the agents are reasoning about. Each probe annotated with what it proves AND what its output would look like if blue/green is UNSAFE.

**Synthesis** produces a 3-column verdict table + one-sentence recommendation: SHIP / DEFER (until probes confirm) / REPLACE with pm2 reload / INSTALL staging-subdomain first. **Verdict filled in below when workflow returns.**

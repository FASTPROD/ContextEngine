### C. Audit-chain race fix — resolves `audit-001-write-race` (commit `8e3e5c1`)

The "chain broken at 2826" carry-forward (Sessions 11-13) RESOLVED.

**Root cause**: appendAudit() had two race surfaces:
1. cachedLastHash was process-local; multiple processes each cached the same stale value before any wrote.
2. appendFileSync is atomic for small writes, but the read-prev-hash → compute-hash → write window was unsynchronized across processes.

**Fix** (LOCK `[AUDIT-001-WRITE-RACE-FIX]`):
- File-lock primitive: create `~/.contextengine/audit.lock` via `O_CREAT|O_EXCL|O_WRONLY` (atomic on POSIX + NTFS). Hold for entire read-then-write window. Release on finally.
- Stale-lock recovery: lockfile older than 10s = orphan, force-unlink, retry.
- Synchronous-but-non-spinning wait via `Atomics.wait` on SharedArrayBuffer (5ms poll, 2s total deadline).
- Lockfile contents: PID + timestamp for debugging.
- Cache invalidation: track cachedSize; if statSync(path).size differs at next acquire, re-read prev_hash from disk.

**Verified**:
- All 19 existing audit tests still pass.
- NEW test "concurrent-writer race (audit-001-write-race fix)" spawns 2 Node child processes via execFileSync, each fires 20 appends in parallel via Promise.all against the same audit.log. Without the lock: chain breaks at first interleave. With the lock: `verifyChain().ok === true`, `verifyChain().total === 40`, both worker A's 20 records and worker B's 20 records present.
- Full vitest suite: 197/197 passing.

**Does NOT retroactively heal index 2826** — by design. Chain is supposed to be tamper-evident; healing would defeat the property. Operators who need a clean anchor should rotate the log. `opscontext audit-rotate` is a follow-up.

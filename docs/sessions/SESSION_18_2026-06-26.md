### E. 0.11.1 → 0.11.2 — scoreHtml hang root-cause + fix

Immediately after the 0.11.1 verify, user reported status bar stuck on `OpsContext: Generating HTML score report for INVOK.fr…` with no way to dismiss — 7+ minutes hung, click doing nothing meaningful.

**Root-cause analysis** (audit log `~/.contextengine/audit.log` showed the scoreHtml emit at `17:14:26.738Z`, then no follow-up):

1. `runCLI()` in [vscode-extension/src/contextEngineClient.ts:119-153](../../vscode-extension/src/contextEngineClient.ts#L119) used `execFileAsync` with default `killSignal: "SIGTERM"`. When the 60s timeout fires, SIGTERM goes to the npx wrapper. The npx wrapper can ignore SIGTERM after its grandchild (the actual `opscontext` binary) has exited, leaving stdio pipes open. `execFileAsync` never resolves because the child process pipes never close. Promise pending FOREVER.
2. `scoreHtml` command in [vscode-extension/src/extension.ts:681](../../vscode-extension/src/extension.ts#L681) used `cancellable: false` on `vscode.window.withProgress(...)`. Even when user notices the hang, there's no X button. Only escape = `Developer: Reload Window`.

**Fix shipped:**

- `runCLI` now sets `killSignal: "SIGKILL"` (non-ignorable; the process MUST exit, pipes MUST close).
- `runCLI` now accepts an `AbortSignal` as an option; propagates to `execFile.signal`.
- `runCLI` re-throws `AbortError` distinctly from real failures so callers can distinguish "user cancelled" from "operation failed."
- `generateHtmlScoreReport(projectName, signal)` accepts the new optional signal parameter; passes through.
- `scoreHtml` command in `extension.ts` now uses `cancellable: true` + wires `token.onCancellationRequested(() => abortController.abort())` + silently exits on AbortError (no error popup for user-initiated cancellation, only for genuine failures).

Net effect: user can click the X on the progress notification at any time → subprocess killed cleanly → no popup → fresh state. Hangs > 60s also auto-kill via `killSignal: "SIGKILL"` + timeout.

## Meta-note — pattern that emerged this session

Three of the user's questions ("17 MCP tools?", "INVOK.fr stuck?", "version alignment?") were not bug reports in the strict sense — they were *observations* by the user that something was off. Each one became a structural fix:

- "17 MCP tools" → 2 hardcoded constants found + replaced with a single source of truth + regression test → can't drift again
- "INVOK.fr stuck" → root cause was a generic class (execFileAsync default SIGTERM) → fix benefits all 6 callsites of runCLI, not just scoreHtml
- "version alignment?" → captured as a backlog entry with a clear trigger (extension v1.0 maturity) → won't get re-asked

This is the right shape of work: each observation triggers a structural change rather than a band-aid. The session shipped 7 commits + 3 publishes + 3 tags in ~3 hours, with each change covered by either a test, a regression guard, or a clear CHANGELOG entry. No "fix forgotten" risk. The pattern is the inverse of the bug class that this session itself proved: when something needs to be updated in two places, force-couple them so future-you can't forget.

## Meta-note on this session's correction cycle

Both the SESSION_16 postscript AND the original `feedback_summary_can_lie.md` framed Session 16 as "the summary fabricated workflow wuf7jr1st." That framing was itself wrong. The correction cascade was:

```
SESSION_16 doc says (initial)        : "wuf7jr1st workflow in flight"
SESSION_16 postscript says (mid-resume): "wuf7jr1st was fabricated — no workflow ran"
SESSION_17 (this doc) says (corrected) : "wuf7jr1st was real all along — I checked the wrong dir + invented a regex"
```

The pattern is its own lesson: when you're wrong about being wrong, the temptation is to rush a single new lesson out. Better to verify the corrected story across at least one independent surface (here: the wuf7jr1st.output file's content matching 07fd98d's diff) before writing the lesson. Done that way around, the lesson is durable; done the other way, the lesson is just the next thing to retract.

The new CLAUDE.md block + the corrected memory file capture both layers of this — *verify before declaring fabricated* AND *if you find yourself doing a correction-of-the-correction, slow down and verify the new story too*.

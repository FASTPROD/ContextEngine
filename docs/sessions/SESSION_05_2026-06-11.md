### Ed25519 production deploy (the big one) — and the path bug caught in production

I deployed in steps rather than running `server/deploy.sh` as-is because the script had latent issues (server-side `tsc` needed devDeps that `npm install --production` omits, and bare `pm2` doesn't resolve in non-interactive ssh). Streamlined deploy:

1. `rsync` `server/` to `/var/www/contextengine-server/` (excluding `node_modules/`, `dist/`, `data/`, `delta-modules/`). **The private key at `server/.secrets/ed25519-license-private.pem` rode along correctly** with mode 0600 preserved by `rsync -a`.
2. `rsync` main `dist/` to `/var/www/contextengine-dist/` for the gen-delta step.
3. Local `npm run build` in `server/` to produce a clean compiled `dist/`, then `rsync` that to the server.
4. PM2 restart via `/usr/local/node-v18.19.0-linux-x64/bin/pm2 restart contextengine-api`.

**First restart attempt failed in production.** The error log showed:

```
Error: Ed25519 private key not found at /var/www/.secrets/ed25519-license-private.pem.
```

**Root cause**: `server/src/license-sig.ts` `loadPrivateKey()` used `join(__dirname, "..", "..", ".secrets", ...)`. From compiled `dist/license-sig.js`, two `..` walks up to `/var/www/` instead of `/var/www/contextengine-server/`. I confused source-tree semantics (`server/src/...` needs one `..` to reach `server/`) with compiled semantics (`server/dist/...` also needs one `..`). PM2 hit max restarts and the activation server went 502 for a few minutes.

**Fix**: changed `..", ".."` to a single `".."`. Local rebuild → rsync the fixed `license-sig.js` + `.map` → PM2 restart. Boot log on second restart:

```
✅ Ed25519 license-signing key loaded
🔑 ContextEngine Activation Server
   Port: 8010
   DB: /var/www/contextengine-server/data/licenses.db
   Delta modules: 3 loaded
   Ready.
```

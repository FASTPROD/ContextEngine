### C. Ed25519 production deploy + the path bug caught in prod
The big one. `server/src/license-sig.ts loadPrivateKey()` used `join(__dirname, "..", "..", ".secrets", ...)` — from compiled `dist/license-sig.js` that walks up to `/var/www/` instead of `/var/www/contextengine-server/`. PM2 hit max restarts; `/health` returned 502 for ~3 minutes.

**Detection**: came directly from CLAUDE.md's "DEPLOY = VERIFY LIVE" rule — I ran `curl https://api.compr.ch/contextengine/health` immediately after the restart and saw the 502.

**Fix**: one `..` removed. Local rebuild + rsync the fixed `dist/license-sig.js` + PM2 restart. Boot log post-fix: `✅ Ed25519 license-signing key loaded`. All 4 DB licenses + 1 activation preserved through the restart cycle.

**Annotation**: the bug is documented at the call site with a comment so a future cleanup pass doesn't reintroduce the two-`..` pattern.

- **`opscontext install-skill [--global | --project] [--force]`** — copies `skills/opscontext/SKILL.md` into Claude Code's skills directory. Installed both project-local (in this repo's `.claude/skills/opscontext/`) and global (in `~/.claude/skills/opscontext/`). Claude Code surfaced the `opscontext` skill in the available-skills list immediately after install.

### E. npm publish — three tokens before success
The original granular token (`ContextEngine-Publish-GranularApr2026`) had expired according to the npm UI but the metadata file said it was still valid until July. The first publish attempt returned a 404 PUT (npm's misleading code for "your token can't write here"). Token #2 (scope-level write, **no** bypass-2FA) hit a 403 because publishing requires either an OTP at publish time or the bypass-2FA flag on the token. Token #3 (`compr-scope-publish-2026-06` — scope-level write **+** bypass-2FA) worked.

The token-creation guidance went through a small workflow that fetched the official npm docs and verified the exact form labels before answering — turned out to be the right call because the UI does NOT have a "Classic Tokens" section to scroll down to (the user corrected my earlier guidance).

`@compr/opscontext-mcp@2.0.0` published. `@compr/contextengine-mcp` deprecated with a pointer at the new package. Both verified via direct registry GET.

### F. `server/deploy.sh` rewritten
The original was brittle: bare `pm2` (not on non-interactive ssh PATH), server-side `tsc` (fails on missing devDeps), no post-deploy verification. New version has five phases — pre-flight (local build + SSH + key checks) → rsync → runtime deps → gen-delta → PM2 restart with full path → **smoke test that polls `/health` for up to 30 s and DIES if it never reports healthy**. `--dry-run` supported. Tested via dry-run against the live server — pre-flight passed cleanly.

- Bumped to 2.0.1 (semver-patch — tightening of a check that was documented as temporary from day one).

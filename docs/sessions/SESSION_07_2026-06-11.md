## What was done

### A. Activation guidance for the user's other VS Code window
User had a second VS Code window still using the cached 2.0.0 npx tarball. Gave the one-liner:

```bash
rm -rf ~/.npm/_npx && echo "now Cmd+Shift+P → Developer: Reload Window in the other VS Code"
```

`rm -rf ~/.npm/_npx` wipes the npx cache so the next call to `npx -y @compr/opscontext-mcp` downloads 2.0.1 fresh from the registry. Doesn't affect any project, npm globals, or anything else. User ran it from FC_project's terminal — fine, the cache path is global.

### B. Workflow `wf_df4b7838-1f8` — VS Code extension audit + plan
Two phases. Phase 1 mapped every "ContextEngine" or "CE" string in `vscode-extension/` to file:line with a recommended action (rebrand_now / rebrand_eventually / keep_as_ce_for_back_compat). Phase 2 picked a publishing strategy and turned the audit into a 23-step ordered action list + 12-command publish sequence + rollback plan.

**Strategy A chosen** (rebrand in place):
- KEEP: extension ID `css-llc.contextengine`, all 6 command IDs (`contextengine.*`), all 5 config keys (`contextengine.*`), chat participant ID + `@contextengine` handle.
- CHANGE: displayName, description, README, status bar text, output channel name, all user-visible notifications, info panel HTML, chat participant fullName, CLI delegate npm package name.
- Rationale: changing the Marketplace ID would force every user to manually reinstall and lose keybindings/settings. Splitting the listing splits review stars and install counts. Strategy A respects all four hard constraints and the rebrand lands without breaking anyone.

- ✅ `~/.copilot-credentials.md` "VS Code Marketplace" section exists (PAT documentation entry present).
- ✅ `@compr/opscontext-mcp@2.0.1` confirmed live on npm registry.

- ✅ Safety tag `pre-rebrand-0.7.1` created on the pre-rebrand HEAD for rollback.

### D. Edits applied — 23 changes across 10 files
- **`vscode-extension/package.json`** — displayName → "OpsContext — AI Agent Compliance", description with OpsContext, version 0.7.1 → 0.8.0, repository URL fixed (was pointing at npm; now points at GitHub), added explicit `homepage` field, keywords reordered to lead with `claude-code, ai-agents, ops, audit-log, compliance, policy-as-code`. All 6 command titles → "OpsContext: …" (command IDs KEPT). Chat participant fullName → "OpsContext" (id + name KEPT). configuration.title → "OpsContext" (property keys KEPT).
- **`vscode-extension/src/contextEngineClient.ts`** (LOCK `[OPSCONTEXT-CLI]`) — CLI delegate switched from `@compr/contextengine-mcp` to `@compr/opscontext-mcp`. Without this, every PRO/search/sync feature in the extension would silently fall back to a 1.x install once the deprecated npm package stops being installable.
- **`vscode-extension/src/{extension,statusBar,chatParticipant,notifications,outputLogger}.ts`** — bulk sed replaced user-visible "ContextEngine" with "OpsContext" and `@compr/contextengine-mcp` with `@compr/opscontext-mcp`. Internal identifiers (variable names like `rawChannel`) left as-is.
- **`vscode-extension/src/infoPanel.ts`** — same user-visible rebrand, PLUS the hardcoded `v0.6.0` footer (which had drifted from the actual 0.7.1) is now read dynamically from `vscode.extensions.getExtension("css-llc.contextengine")?.packageJSON?.version` so the drift can never recur.
- **`vscode-extension/README.md`** — full pass with a top-of-page note: *"Previously published as ContextEngine — same extension, same install ID, same `@contextengine` chat handle, same settings keys. Auto-update brings you the OpsContext rebrand."*
- **`vscode-extension/CHANGELOG.md`** — new `[0.8.0] — 2026-06-11 — Renamed to OpsContext` section listing what changed and what's preserved.

File-header docstrings that reference "ContextEngine Client" / "ContextEngine knowledge base" were intentionally left alone — those describe historical file purpose and aren't user-visible.

### E. Compile + package + publish
- `rm -rf dist && npm run compile` — clean tsc.

- `vsce ls-publishers` confirmed `css-llc` credentials are in macOS Keychain (no need to manage PAT manually).
- `npx @vscode/vsce publish 0.8.0` → published successfully.

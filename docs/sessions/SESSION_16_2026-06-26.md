Three deploy.sh bugs surfaced in the first real-run + fixed in this session:

| # | Bug | Fix | LOCK |
|---|---|---|---|
| 1 | Whitelist included phantom `assets/css/style.css` (CSS is inline) → pre-flight refused to proceed | Removed entry; comment explains inline-CSS design | — |
| 2 | Single-shot 2s wait for CDN cache too short → post-deploy marker false-negative | Retry-with-backoff loop, 8 attempts × cumulative ~55s + Cache-Control headers | (in commit) |
| 3 | `set -o pipefail` + `curl -sf "$URL" \| grep -q "$MARKER"` is a 100% false-negative trap — grep-q exits 0 on first match → closes stdin → curl gets SIGPIPE → exits 56 → pipefail propagates non-zero → if-test sees failure even though grep matched 7 times | Replace pipe with `curl -o tmpfile + grep tmpfile` (no pipe → no SIGPIPE → no pipefail interaction) | `[COMPR-FR-DEPLOY-PIPEFAIL-TRAP]` |

The pipefail trap was the most insidious — reproduced empirically with `bash -c 'set -o pipefail; curl -sf "$URL" | grep -q "$MARKER"; echo $?'` returning 56 even when curl + grep both succeed in isolation. Saved as persistent learning under `devops` category: any script that does `curl | grep -q "marker"` inside `set -o pipefail` will false-fail.

- **P0 (critical correctness)**: pre-commit hook reads `.git/COMMIT_EDITMSG`, but git does NOT populate that file before pre-commit runs. First commit reads empty string → false positive (blocks valid commit). Subsequent commits read the PREVIOUS commit's message → false positive AND false negative AND the WRONG message is recorded in the audit log. **The test suite hid this** by pre-writing the file. The correct git hook for this gate is `commit-msg` (runs after message is written, gets path as $1).
- **14 bypass surfaces** documented in the verifier output. Top 3 to address: (#3) regex matches the literal bypass marker text; (#5) commented-out bypass marker passes; (#11) bypass-reason min-length 5 chars with no semantic check.

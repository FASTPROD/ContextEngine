## What was done

### A. Claude Code hook — terminal-side capture closed

`~/.claude/hooks/opscontext-emit.sh` (bundled in repo at [defaults/claude-code-hook.sh](../../defaults/claude-code-hook.sh)). LOCK `[OPSCONTEXT-CC-HOOK]`. Pure bash + jq + curl. ~28 ms per invocation.

Hook events wired in `~/.claude/settings.json`:
- `UserPromptSubmit` → emits `vscode.prompt_submit` (feeds the `loop` heuristic)
- `PostToolUse` matcher `.*` → emits `vscode.tool_call` (feeds `stuck`, `silent_failure`)
- `SessionStart` → emits `vscode.session_start`

Critical discipline: **PostToolUse only, not PreToolUse**. Emitting both would double-count for the `stuck` heuristic and skew `silent_failure` counts. Documented in the LOCK.

Namespace choice: **`vscode.*` not `claude_code.*`**. The published 2.1.0 detector heuristics already read `vscode.*` events; namespacing under it lets the heuristics fire **today** without a parallel namespace migration. `payload.surface = "claude-code"` disambiguates for any caller that cares.

Transport: HTTP `POST /events`, NOT direct file writes. Keeps event writes serialized through the running MCP server's single in-process chain cache — sidesteps the historic concurrent-write race we found earlier today.

Safety: **silent on every failure**. Server down, secret missing, jq error → exit 0, never block Claude Code. 1 s hard timeout on the HTTP call.

E2E verified: 5 simulated PostToolUse calls → audit log → `watch --once` fired **stuck** (WARN) + **silent_failure** (CRIT), exit code 2.

### B. macOS LaunchAgent — install-autostart

`~/Library/LaunchAgents/com.opscontext.mcp.plist`. Per-user (`gui/$UID`), no root. RunAtLoad + KeepAlive + ThrottleInterval 10. Logs to `~/.contextengine/logs/mcp-{stdout,stderr}.log`.

Boot flow: `launchctl bootout` (idempotent) → `launchctl bootstrap gui/$UID <plist>`. Wait up to 30 s for port 7842 to bind, then `curl /health` to confirm.

For the user's local environment, plist points at the dev tree (`/Users/yan/Projects/ContextEngine/dist/index.js`) since he's the dev. For other users, the new `install-autostart` CLI auto-detects either a global npm install (via `npm root -g`) or a co-located `dist/`.

## Mid-day live-verification arc (afternoon, after the 2.1.1 commit + push)

The maintainer ran the Day-1 test plan from `/Users/yan/Projects/FC_project` (a project unrelated to ContextEngine, to simulate a real user). Three bugs surfaced + fixed within ~90 minutes:

1. **`commit 99d57ad` — fix(chrome-ext): bundle content scripts as IIFE**. The 0.1.0 chrome ext shipped `dist/content/claude.js` with top-level `import` statements. Chrome MV3 content_scripts don't support ES modules; the manifest can put `"type":"module"` on `background` but not on `content_scripts`. Result: SyntaxError on line 13, script never ran. Fix: `scripts/bundle-content.mjs` runs esbuild with `format: "iife"` after `tsc` to inline all `./shared/*` imports.
2. **`commit 5206626` — fix(chrome-ext): patch claude.ai selectors + switch prompt capture to rendered DOM**. Two distinct bugs in one fix:
   - **Race**: `attachPromptListeners` did `setTimeout(capturePromptSubmit, 0)` on Enter — React cleared the input before our deferred read ran. Replaced the input-intercept path entirely; now captures from the rendered `[data-testid="user-message"]` DOM nodes via the same debounceSettle observer that handles responses.
   - **Selector drift**: Anthropic shipped a UI refresh between 2026-05-24 and 2026-06-23. Updated `assistantMarkdown` (`font-claude-message` → `font-claude-response`), `streamDoneMarker` (`aria-label="Copy"` → `data-testid="action-bar-copy"`), added new `userMessage` tier.
3. **`commit 7dcf0d2` — fix(chrome-ext): broaden response stream-done check to document-wide**. Per-turn scoping (`block.closest('div[data-testid^="conversation-turn-"]...')`) failed because Anthropic dropped the `conversation-turn-*` testids. Replaced with a doc-wide `anyDone` check: if any `action-bar-copy` button exists anywhere in the document, latest stream is done, emit any unseen assistant blocks.

After 7dcf0d2, end-to-end verified: prompts + responses captured live on claude.ai/new with the maintainer typing "what is 2+2" and getting "4 — same as a moment ago. Is something not adding up on your end..." landing in the audit log within 750ms of stream completion.

### Known polish: streaming response over-emission

One observation under live test: a single response can fire ~6 `browser.response` events during streaming because debounceSettle (750 ms) fires multiple times as the response grows; each settle has a different `text.length`, so the dedupe key differs, and the same response gets re-emitted. Not blocking. Filed as `chrome-ext-001-streaming-overemit` for the next session — fix candidates: per-block lastEmittedLength tracking, longer debounce dedicated to response settles, or hybrid stability detection. Workflow design + adversarial verify in flight.

## Findings worth remembering

### 🟡 Audit-log concurrent-write race (filed but historic)
Earlier today, `audit-verify` flagged the global log as BROKEN at index 2826. Investigation: 8 chain breaks total, all clustered on **2026-06-10 / 06-11**, all `system` actor, all `activation.legacy_signature` ↔ `session.save` racing each other on the same millisecond. The activation server (writing `legacy_signature` events at boot) was concurrent with the main MCP server (writing `session.save`); both read `prev_hash` before either wrote, then both wrote with stale data. **Zero new breaks since flag-day (2026-06-11)** because `legacy_signature` events stopped after Ed25519 cutover. The race window still exists in principle between any two concurrent writers — but with HTTP-only writes through the launchd-managed MCP server, all events go through ONE in-process chain cache, sidestepping it. Real fix: `flock` on append in `src/audit.ts`. Filed as `audit-001-write-race` for a future session.

### 🟢 28 ms hook latency is acceptable
Measured: 10 sequential PostToolUse invocations = 281 ms total (~28 ms each). Bash + jq + curl is the right toolchain for this; Node startup (~80 ms) would have added 50 ms+ per tool call and become user-visible.

### 🟢 LaunchAgent boot pattern matches the user's existing ecosystem
`com.invocme.backup-{nightly,weekly}.plist` already in `~/Library/LaunchAgents/`. Using the same idiom keeps mental model consistent.

## Open questions

1. **Should `install-claude-hook` also re-namespace events under `claude_code.*` when 2.2.0 lands?** Pros: cleaner attribution. Cons: requires bumping the http-server.ts allowlist + bumping the detector to read both `vscode.*` and `claude_code.*` for transition. **Recommendation: defer.** The `payload.surface = "claude-code"` field is enough disambiguation for now; revisit if/when the heuristics need source-specific tuning.

2. **Should the LaunchAgent restart on `npm install -g @compr/opscontext-mcp@latest`?** Currently the plist pins an absolute path so an upgrade requires re-running `install-autostart --force`. Could add a post-install npm script that detects + refreshes the plist. **Recommendation: defer.** First confirm `install-autostart` works for real users; auto-refresh is a polish step.

3. **Menu-bar app for visual liveness signal?** The user persona ("vibe coders") would benefit from a tiny menu-bar indicator showing green/amber/red based on detector signals. ~1 week of Swift work. **Recommendation: validate Phase 1 usage first.** No point building UI for a feature nobody uses yet.
